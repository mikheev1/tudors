# Архитектура и развитие Tudors Booking 360

## 1. Цель продукта

Сервис должен решать три задачи одновременно:

- показывать площадку так, будто пользователь по ней проходит;
- позволять быстро сравнить цены, вместимость, стиль и услуги;
- превращать просмотр в бронирование без ручной переписки.

## 2. Архитектура по слоям

### Frontend

- Landing и каталог: `Next.js`
- Личный кабинет клиента: `Next.js`
- Личный кабинет партнера: `Next.js`
- Viewer: `Marzipano` или `Three.js`
- Карты и floor plan: SVG overlay или canvas

### Backend

- API gateway: `NestJS`/`Fastify`
- Auth: JWT + refresh tokens
- Booking service: слоты, блокировки, подтверждения
- Media service: загрузка и публикация панорам
- Tour service: сцены, хотспоты, маршруты
- Billing service: предоплата, возвраты, счета
- Notification service: Telegram, SMS, Email

### Data

- `PostgreSQL` для основных данных
- `Redis` для кэша и краткосрочных блокировок
- `S3` для медиафайлов
- `ClickHouse` или `BigQuery` позже для аналитики

## 4. Как расти без боли

### MVP

- Один сайт.
- Один viewer.
- Ручное подтверждение менеджером.
- Без сложной оплаты.

### Growth

- Поиск по нескольким городам.
- Реальная доступность по календарю.
- Кабинет площадки.
- Интеграция с CRM.

### Scale

- Мультиарендность.
- Роли: владелец, менеджер, оператор контента, супер-админ.
- Версионирование туров.
- Контент-пайплайн через очередь задач.

## 5. Отдельный пайплайн для 360/3D

Это важно: медиа-часть должна жить отдельно от обычного CRUD.

Поток такой:

1. Съемка объекта.
2. Stitching и ретушь.
3. Контроль качества.
4. Загрузка master файла.
5. Генерация тайлов и нескольких разрешений.
6. Создание сцен и хотспотов.
7. Публикация новой версии тура.

Почему так:

- тяжелые файлы нельзя обрабатывать синхронно из пользовательского запроса;
- нужен контроль версий;
- иногда одну и ту же локацию переснимают несколько раз в год.

## 6. Как сделать "ходить по месту"

Есть три уровня реализации:

### Вариант A. Equirectangular 360 + hotspots

Самый реалистичный путь на старте.

- Снимаются 360-панорамы.
- Между точками делаются переходы.
- Пользователь нажимает точки и переходит дальше.

Плюсы:

- быстро;
- дешево;
- хорошо работает на телефонах.

### Вариант B. 3D mesh + panoramas

- Делается базовая 3D-модель пространства.
- Панорамы привязываются к координатам в модели.
- Можно показывать floor map и более естественные переходы.

Плюсы:

- лучше навигация;
- можно строить путь по этажам.

Минусы:

- дороже пайплайн;
- сложнее поддержка.

### Вариант C. Полноценный digital twin

- LiDAR/фотограмметрия;
- 3D-сцена;
- интерактивная расстановка мебели и декора.

Это уже следующий этап, не MVP.

- глубина прохождения тура;
- среднее время в туре;
- конверсия из заявки в предоплату;
- процент объектов с актуальным туром;
- скорость публикации нового объекта.
